# `src/` 代码规则

这些规则适用于 `src/**`。

## 实现原则

- 先找对应设计文档，再实现
- 优先做可测试、可替换的最小模块
- 玩法逻辑和表现层尽量解耦
- UI 不直接拥有玩法状态
- 热路径里避免明显无意义分配和重复查询

## 玩法代码

- 关键玩法数值不要散落硬编码
- 能放进 `assets/data/` 的配置，优先外置
- 如果暂时必须硬编码，写清楚为什么是临时方案，以及未来要迁到哪里
- 状态切换、伤害结算、冷却、生成规则等逻辑要尽量纯函数化或模块化

## 改动同步

- 新玩法系统至少补一个对应设计文档或设计草稿
- 改架构时补 ADR
- 改数据结构时同步检查 `assets/data/`
- 改纯逻辑时优先补测试

## 提交前自检

- 这次实现解决的是不是最小可玩的切片
- 有没有把 UI、资源加载和玩法规则写死耦合在一起
- 有没有遗漏边界条件、失败路径、空数据路径

# 独立游戏工作室精简版

这份文档把“Claude Code Game Studios”那种重型组织，压缩成适合独立开发者的轻量模型。

## 为什么要精简

独立开发最怕的不是“没有流程”，而是“流程比产出还重”。
所以这个版本只保留真正常用的能力：

- 帮你明确下一步
- 帮你把模糊想法变成可实现规则
- 帮你控制范围
- 帮你补验证和复盘

## 8 个核心工作模式

| 模式 | 什么时候用 | 产出重点 |
| --- | --- | --- |
| 概念探索 | 还不知道做什么，或概念太散 | 核心乐趣、题材、目标玩家 |
| 系统设计 | 知道要做什么，但规则不完整 | 系统文档、边界条件、调参项 |
| 技术决策 | 不确定架构、引擎组织、数据流 | ADR、权衡、风险 |
| 核心实现 | 已有规格，准备做最小可玩切片 | 代码、数据、测试 |
| 内容与界面 | 需要信息表达、手感反馈、引导 | UI 文案、交互、可读性 |
| 代码评审与 QA | 想查 bug、回归、缺测 | Findings、风险、补测建议 |
| 计划推进 | 任务太散、不知道先做什么 | Sprint、优先级、范围裁剪 |
| 打磨与发布准备 | 已经可玩，准备拉体验和稳定性 | 性能、流程、发布检查 |

## 48 个角色为什么压缩成 8 个模式

原版的强项是“分工完整”，但独立开发更需要“切换顺手”。
你并不真的需要 48 个人格，而是需要 8 种稳定视角：

- 创意视角
- 设计视角
- 技术视角
- 实现视角
- 体验视角
- 审查视角
- 制作人视角
- 收尾视角

## 推荐默认节奏

### 从 0 开始

1. 概念探索
2. 系统设计
3. 技术决策
4. 计划推进
5. 核心实现
6. 代码评审与 QA

### 已经有项目

1. 计划推进
2. 代码评审与 QA
3. 系统设计或技术决策
4. 核心实现
5. 打磨与发布准备

## 轻量化原则

- 不强迫每个任务都走完整流程
- 不要求每个系统都先开会再写
- 只在风险高、范围大、信息不足时放慢
- 默认先做最小闭环，再扩展

## 你真正需要维护的只有 4 类东西

1. `production/studio-status.md`
2. `design/gdd/` 里的关键系统设计
3. `docs/architecture/` 里的重要技术决策
4. 可运行的代码和最关键的测试

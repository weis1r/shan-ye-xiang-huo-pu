# Codex Game Studio Lite - 根规则

你正在一个面向独立游戏开发的轻量工作室模板中工作。
默认使用中文与用户沟通，除非用户明确要求英文。

## 核心目标

- 帮用户把游戏从想法推进到可玩版本
- 保持结构化协作，但避免过重流程
- 在设计、实现、验证之间保持同步

## 先读什么

开始任何任务前，按需读取这些文件：

1. 当前目录下的本文件
2. 与目标路径对应的分目录 `AGENTS.md`
3. `production/studio-status.md`
4. 相关设计文档、冲刺文档、ADR 或测试文件

## 默认工作方式

### 1. 先理解上下文

- 先扫描项目结构，再动手
- 如果任务涉及已有系统，先读相关设计文档和代码
- 如果缺少关键上下文，优先提出 2 到 3 个聚焦问题，或者给出 2 到 3 个选项供用户决策

### 2. 用合适的工作模式思考

根据任务自动切换到最接近的模式：

- 概念探索：找玩法、主题、核心乐趣
- 系统设计：拆系统、补规则、收边界
- 技术决策：选架构、定约束、控风险
- 核心实现：做最小可玩的那一层
- 内容与界面：服务可读性、反馈和体验
- 代码评审与 QA：优先找 bug、回归和测试缺口
- 计划推进：拆优先级、排 sprint、控范围
- 打磨与发布准备：性能、手感、可访问性、上线检查

### 3. 先说计划，再做大改

如果要改多个文件、创建新目录或重构已有结构：

- 先用 1 到 2 句话说明你准备改什么
- 明确会涉及哪些文件
- 优先做最小闭环，而不是一次性铺满整套系统

### 4. 设计与实现同步

- 新功能如果改变玩法、规则、资源结构或流程，更新对应设计文档
- 重要技术决策写进 `docs/architecture/`
- 如果代码已经明显偏离设计文档，先指出，再决定是改代码还是改设计

### 5. 验证是默认动作

完成改动后，默认执行：

```bash
bash scripts/studio-check.sh
```

如果项目里以后加入了引擎测试、构建或 lint 命令，也应一起运行。

## 与用户协作的规则

- 用户是最终决策者
- 遇到范围膨胀时，主动给出更小切片的建议
- 不在未说明的情况下擅自生成大量无关文件
- 不提交 commit，除非用户明确要求
- 复盘时优先说结果、风险、验证状态和下一步

## 目录约定

- `src/`：代码
- `assets/data/`：数据配置
- `design/gdd/`：玩法和系统设计文档
- `docs/architecture/`：技术决策与架构说明
- `production/`：进度、里程碑、状态
- `prompts/`：可直接复用给 Codex 的提示卡

## 做得更像独立工作室，而不是更像流程公司

- 偏向一周内能见到成果的方案
- 偏向先验证风险最高的部分
- 偏向低维护成本而不是“架构显得很高级”
- 偏向能玩、能测、能迭代，而不是文档过量
